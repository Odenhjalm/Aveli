# CMT-003.5_CANCEL_INTENT

- TYPE: `OWNER`
- TITLE: `Add non-authoritative membership cancel intent while preserving webhook-only state authority`
- DOMAIN: `membership cancellation intent`
- GROUP: `WEBHOOK REPAIR / INTENT`

## Problem Statement

The contract forbids treating portal or cancellation surfaces as launch purchase authority, but it does not forbid a backend-owned cancellation intent surface when that surface remains non-authoritative. The repo needs a way for an authenticated user to express cancellation intent to Stripe without mutating `app.memberships` directly and without bypassing webhook settlement.

## Contract References

- `commerce_membership_contract.md` sections `1`, `3`, `5`, `6`, `8`, `10`, `13`

## Audit Inputs

- `AUD-05` undeclared billing-management surfaces
- `AUD-07` webhook must remain the only canonical payment-confirmation and membership-mutation path

## Implementation Surfaces Affected

- `backend/app/routes/billing.py`
- `backend/app/services/subscription_service.py`

## DEPENDS_ON

- `CMT-003_WEBHOOK_REPAIR`

## Purpose

Allow a user to express cancellation intent in backend while preserving Stripe webhook processing as the only path that may change canonical membership state.

## Required Behavior

- Add `POST /api/billing/cancel-subscription-intent`.
- The endpoint validates the authenticated user.
- The endpoint resolves the user's Stripe subscription reference from backend-owned records.
- The backend sends the cancellation intent to Stripe by canceling immediately or setting `cancel_at_period_end`, according to the final implementation choice.
- The endpoint records only intent-side operational data needed for observability and follow-up.

## Explicit Non-Authority Rules

- The endpoint MUST NOT mutate `app.memberships`.
- The endpoint MUST NOT infer or project canonical membership state from the Stripe response.
- The endpoint MUST NOT grant access.
- The endpoint MUST NOT revoke access.
- Membership state transition happens only later through canonical webhook handling.

## Acceptance Criteria

- Cancel intent triggers a Stripe-side cancellation change.
- No direct membership mutation occurs in the cancel-intent endpoint or service path.
- Backend response is treated only as intent acknowledgment, not authority.
- A later Stripe webhook remains responsible for any resulting membership lifecycle transition.

## Stop Conditions

- Stop if the implementation requires direct mutation of `app.memberships`.
- Stop if the Stripe response must be treated as current-state authority to complete the flow.
- Stop if access changes would occur before webhook processing.

## Out Of Scope

- Frontend authority changes
- Direct access revocation logic
- Contract edits
- Reclassification of existing tasks
